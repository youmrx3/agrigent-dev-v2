## 4. WHAT DATA SHOULD NEVER BE SAVED

### ❌ String-formatted values with units
```sql
-- WRONG — Do NOT save display strings in the DB:
moisture = '68%'        -- save as NUMERIC 68
npk = '45 mg/kg'        -- save as NUMERIC 45
temperature = '24°C'    -- save as NUMERIC 24
ph = '6.8'              -- save as NUMERIC 6.8
```
**Why:** Formatting (units, symbols, rounding) is a UI concern. Saving raw numbers lets you do aggregations (`AVG`, `SUM`, `MIN`, `MAX`) and change display format later without data migration.

### ❌ Duplicated computed data
```sql
-- WRONG — Don't store what can be derived:
profileName = 'Wheat - Flowering'  -- This is a JOIN of profiles.crop_type + profiles.growth_stage
```
**Why:** Store `profile_id`, then use a `JOIN` or the display logic in the frontend. Duplication creates sync bugs.

### ❌ UI-only state
```sql
-- WRONG — These belong in localStorage or Zustand, not PostgreSQL:
valueColor = 'text-green-400'
is_read = false on notifications  (actually this one IS fine to store)
```
**Why:** Colors, positions, animation states, and component visibility are UI concerns. The DB should store raw data, not presentation metadata. `is_read` is an exception — it's legitimate business state.

### ❌ Secrets / Credentials
- **Passwords** — Never store. Supabase Auth handles this automatically.
- **API keys** — Use `.env.local` environment variables, not the DB.
- **Session tokens** — Supabase Auth manages these in `auth.sessions`.

### ❌ Hardcoded mock data in the app
**`data/dashboard-data.ts`** — This file is now **deprecated**. It was the Phase 1 mock. The real data lives in Supabase. The file can be deleted once the UI is fully migrated.

## 5. DATA FLOW — How Data Moves Through the System

### 5.1 Signup / New User Flow
```
User fills signup form
  → POST /api/signup (server-side, uses service_role key)
    → supabase.auth.admin.createUser()  (creates user in auth.users)
    → Returns user object to client
  → Client calls supabase.auth.signInWithPassword()
    → Gets a session token
  → seedInitialData() fires:
    → INSERT 2 profiles
    → INSERT 3 sensors
    → INSERT 7 water_analytics rows
    → INSERT 2 reading_history rows
    → INSERT 2 recommendations
    → INSERT 2 notifications
    → UPSERT 1 user_settings
    → INSERT 2 activity_log entries
  → Redirect to /dashboard
```

### 5.2 Dashboard Load Flow
```
User visits /dashboard
  → AuthGuard checks isAuthenticated
  → AuthInitializer restores session from Supabase
  → Each page component mounts:
    → useEffect() calls dashboard-service function
    → Service calls supabase.from('table').select('*').eq('user_id', auth.uid())
    → Supabase RLS filters to current user's rows
    → Service maps snake_case DB → camelCase UI types
    → Zustand store.set() updates state
    → Component re-renders
  → startRealtimeSimulation() runs every 5s (client-side only — does NOT write to DB)
```

### 5.3 The 4-Stage Pipeline Flow
```
STAGE 1 — ACQUISITION
  Sensor (IoT device) reads moisture, NPK, pH, temperature
  → Data sent to server
  → INSERT INTO reading_history (moisture, npk, ph, temperature, sensor_id)
  → UPDATE sensors SET moisture=..., npk=..., ph=..., temperature=..., last_reading_at=now()

STAGE 2 — RECEPTION
  System looks up the active crop profile for this sensor's location
  → SELECT * FROM profiles WHERE user_id = ? AND (most relevant/active)

STAGE 3 — EVALUATION
  Each parameter compared against crop-specific optimal ranges:
    moisture: 60% – 80%
    npk: 40 – 60 mg/kg
    ph: 6.5 – 7.5
  → Determines status: within / below / above
  → Determines deviation: mild / moderate / significant

STAGE 4 — OUTPUT
  → INSERT INTO recommendations (reading_id, parameter, value, status, deviation, corrective_action)
  → UI displays recommendations on /dashboard/recommendations
```

### 5.4 Realtime Simulation (CURRENTLY MOCK — Client Only)
```
Every 5 seconds in the browser:
  → Randomly mutates waterAnalytics, stats, sensors values in Zustand store
  → These changes are DISPLAY-ONLY — they do NOT persist to Supabase
  → UI shows "Live" badge despite being client-side simulation

FUTURE: Replace with Supabase Realtime subscriptions:
  supabase
    .channel('sensors')
    .on('postgres_changes', { event: 'UPDATE', schema: 'public', table: 'sensors' },
      (payload) => setSensors(payload.new)
    )
    .subscribe()
```

| Data | Save to DB? | Where | Why |

| Sensor device metadata | ✅ YES | `sensors` | Core business data |
| Sensor readings (raw numbers) | ✅ YES | `reading_history` | Core business data |
| Latest sensor state | ✅ YES | `sensors` (columns) | Quick access without querying history |
| Evaluation recommendations | ✅ YES | `recommendations` | Core business data |
| Water consumption metrics | ✅ YES | `water_analytics` | Chart data / reporting |
| Field zone health scores | ✅ YES | `field_zones`, `health_metrics` | AI module |
| Irrigation schedules & actuals | ✅ YES | `irrigation_daily` | AI module |

| Activity feed | ✅ YES | `activity_log` | Live dashboard feed |

| Color values like "text-green-400" | ❌ NO | — | UI concern |
| Computed fields like "Wheat - Flowering" | ❌ NO | — | JOIN in query |
| Passwords | ❌ NO | Supabase Auth handles | Security |

## 7. SUPABASE CONFIGURATION

### Auth Settings (from Dashboard)
```
Authentication → Settings:
  - Confirm email: OFF (recommended for dev) — set via /api/signup route bypass
  - Rate Limits: Uses /api/signup server route to bypass public endpoint limits
  - Session duration: 3600 seconds (default)
```

### RLS (Row Level Security)
All user-data tables have RLS enabled with policy:
```sql
USING (auth.uid() = user_id)
```
Exceptions:
- `contact_messages` — anyone can INSERT, only authenticated can SELECT
- `auth.users` — managed by Supabase, do not create RLS policies

### Indexes
12 indexes on foreign keys and commonly queried columns (`user_id + timestamp DESC` patterns).

## 8. TABLE MANAGEMENT — CRUD Lifecycle

### What creates each row

| Action | Table(s) affected | Trigger |
|--------|-------------------|---------|
| User signs up | `auth.users` + seed into `profiles`, `sensors`, `readings`, etc. | `seedInitialData()` |
| User creates profile | `profiles` | Profile creation form |
| User adds sensor | `sensors` | Sensor registration form |
| Sensor reports reading | `reading_history` + `sensors` (update latest) | IoT poll interval |
| System evaluates reading | `recommendations` | After reading insert (trigger or app logic) |
| AI generates prediction | `prediction_events` | AI model cron job |
| System sends notification | `notifications` | Rule engine trigger |
| User performs action | `activity_log` | Action handler |
| User changes settings | `user_settings` (upsert) | Settings save |
| User deletes profile | `profiles` + nullify `reading_history.profile_id` | Profile deletion |
| User deletes account | ALL user's rows + `auth.users` | Account deletion handler |

### What NEVER creates DB rows
- Page navigation
- Chart hover/zoom
- Dark/light mode toggle (localStorage)
- Search input keystrokes (until submitted)
- Mobile menu open/close

## 10. RAW SQL SNIPPETS (Useful Queries)

### Get the latest reading for each sensor
```sql
SELECT DISTINCT ON (sensor_id) *
FROM reading_history
WHERE user_id = auth.uid()
ORDER BY sensor_id, recorded_at DESC;
```

### Count readings in the last 7 days
```sql
SELECT COUNT(*) FROM reading_history
WHERE user_id = auth.uid()
  AND recorded_at > now() - interval '7 days';
```

### Find sensors with no readings in 24h
```sql
SELECT sensor_code, location, last_reading_at
FROM sensors
WHERE user_id = auth.uid()
  AND (last_reading_at IS NULL OR last_reading_at < now() - interval '24 hours');
```

### Average moisture by profile
```sql
SELECT p.crop_type, AVG(rh.moisture) as avg_moisture
FROM reading_history rh
JOIN profiles p ON rh.profile_id = p.id
WHERE rh.user_id = auth.uid()
GROUP BY p.crop_type;
```

### Recommendations with deviation summary
```sql
SELECT status, deviation, COUNT(*) as count
FROM recommendations
WHERE user_id = auth.uid()
GROUP BY status, deviation
ORDER BY status, deviation;
```
